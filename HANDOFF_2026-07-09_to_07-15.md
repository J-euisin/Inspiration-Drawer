## 2. 미해결 / 결과 확인 필요한 작업 (⚠️ 새 대화에서 최우선 확인)

### 2-A. 문장(munjang)/단상(dansang) 카드 텍스트 세로 정렬 불일치 — **가장 시급, 여러 차례 시행착오 중이었음**

**문제**: 같은 3줄 텍스트인데, 문장 카드는 본문이 카드 안에서 세로 중앙 정렬이 잘 되는데, 단상 카드는 출처 영역이 없어서 본문 텍스트 시작 위치가 문장 카드보다 아래로 처져 보임.

**시행착오 순서 (새 대화에서 이 순서를 참고해서 원인을 진단할 것):**
1. 1차 요청: "단상 카드도 문장 카드와 동일하게 본문을 세로 중앙 정렬해달라" → 결과: 문장 카드는 전체적으로 위로 붙어버리고, 단상 카드는 상단 따옴표 아이콘 위치가 오히려 아래로 내려오는 회귀 발생
2. 2차 요청: "따옴표 아이콘은 항상 상단 고정, 본문은 따옴표 아래~출처 위(또는 카드 하단) 사이 공간에서 세로 중앙 정렬" 구조로 재요청 → 결과: 여전히 문장/단상의 텍스트 시작 y좌표가 달랐음
3. 3차 요청: "단상 카드도 출처 영역만큼의 빈 공간을 하단에 확보(레이아웃상으로만, 텍스트는 안 보여도 됨)해서 본문 텍스트 시작점을 문장 카드와 완전히 동일하게 고정"하도록 요청 → **부작용**: 이 방식이 '3장씩' 뷰타입의 line-clamp/말줄임 로직과 충돌해서, 3장씩 보기에서 텍스트가 처음부터 안 보이고 중간부터 잘려 보이는(말줄임표도 사라진) 심각한 회귀 버그가 발생함
4. 4차 요청(가장 최근, 결과 미확인 상태로 대화 종료): "3장씩 뷰타입에는 이 빈 공간 확보 로직을 적용하지 말 것. 3장씩은 애초에 출처가 없으니 본문이 따옴표 아래에서 곧바로 시작해야 함. 동시에 3장씩의 line-clamp(4줄)+말줄임표를 원래대로 복구하되, 폰트 크기를 좀 더 줄이고, 이중 안전장치로 명시적 max-height도 함께 적용."

**→ 새 대화에서 가장 먼저 할 일**: 현재 코드 상태를 스크린샷으로 확인해서, (a) 문장/단상 카드의 텍스트 시작점이 실제로 일치하는지, (b) 3장씩 뷰타입의 말줄임 처리가 정상 복구됐는지 두 가지를 동시에 검증해야 함. 둘 다 동시에 만족시키기가 까다로�웠던 부분이므로, 필요하면 3장씩과 1장씩/2장씩을 완전히 다른 스타일 로직(별도 CSS 클래스 또는 별도 컴포넌트 분기)으로 처리하는 방안도 고려할 것.

### 2-B. iOS 홈 화면 아이콘 미표시
- 문제: 모바일 브라우저에서 "홈 화면에 추가"(바로가기) 시, 안드로이드는 파비콘이 아이콘으로 잘 뜨는데 iOS 사파리는 아이콘이 안 뜸(빈 아이콘/스크린샷 썸네일로 표시됨)
- 원인: iOS는 일반 favicon.ico가 아니라 `apple-touch-icon`이라는 전용 이미지 태그가 head에 있어야 인식함
- 요청한 내용: 180x180px PNG 형식의 apple-touch-icon 파일을 public 폴더에 추가하고, layout.tsx(또는 metadata 설정)에 `<link rel="apple-touch-icon" href="/apple-touch-icon.png" />` 관련 설정 추가, manifest.json의 icons 배열도 확인
- **결과 미확인 상태로 대화 종료. 새 대화에서 반드시 재확인 필요.** (iOS 실기기에서 기존에 추가했던 바로가기를 삭제하고 새로 추가해야 캐시 없이 정확히 확인 가능하다는 점도 감안할 것)

### 2-C. 모바일/PC 공통 입력창 확장 애니메이션 — **미해결, 마지막까지 해결 안 된 채 종료됨**
- 문제: 단상 입력창에 텍스트를 여러 줄 입력할 때, 알약형에서 라운딩 큰 사각형으로 커지는 과정이 부자연스럽게 끊김 (GPT/Gemini 입력창처럼 매끄럽게 늘어나야 함)
- **중요한 실수**: 이 문제는 원래 PC/모바일 공통 문제였는데, Claude가 실수로 "모바일 전용"으로 프롬프트 범위를 제한해서 요청했음. 사용자가 이를 지적함.
- 1차 시도(모바일 전용으로 잘못 제한됐던 요청): textarea에 `transition: height 0.2s ease-out` 추가, `handleInput`에서 `requestAnimationFrame`으로 다음 프레임에 목표 높이 적용, `isMultiline` 판단 임계값을 36px→40px로 조정 → 그래도 PC/모바일 모두에서 여전히 끊김이 남아있었음
- 2차 요청(PC+모바일 공통으로 재요청, 결과 미확인 상태로 종료): `height: auto`로 리셋했다가 다시 목표 px값을 주는 방식 자체가 그 순간 브라우저 리플로우를 유발해서 transition이 걸려있어도 시각적으로 끊겨 보이는 것이 진짜 원인일 가능성이 크므로,
  - (a) 화면에 안 보이는 숨겨진 mirror div에 동일 텍스트를 렌더링해서 필요한 높이를 미리 계산하고, 실제 textarea는 auto 리셋 없이 계산된 값으로 바로 transition을 적용하는 방식, 또는
  - (b) rows 속성 기반으로 높이를 계산하는 방식
  중 하나를 검토해서 적용하도록 요청함.
- **결과(영상/캡처)를 받지 못한 채 대화가 종료됨. 새 대화에서 가장 먼저 결과를 확인하고, 여전히 문제가 있으면 위 (a)/(b) 방식을 실제로 시도해봐야 함.**
